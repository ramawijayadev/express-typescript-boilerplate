Core Feature Handbook

Implementation Handbook & Checklist

Dokumen ini mendefinisikan Core Feature yang disediakan oleh starterkit backend. Tujuannya adalah memisahkan fitur fundamental (Wajib) dari fitur tambahan, serta menjadi pedoman implementasi module yang reusable.

1. Purpose & Scope

Dokumen ini menjadi "blueprint" fungsionalitas aplikasi.

  • Standardisasi: Menentukan feature generik yang harus selalu ada.
  • Reusability: Panduan agar modul inti bisa dipakai ulang di project lain.
  • Prioritas: Memisahkan prioritas pengerjaan (Must Have vs Nice to Have).

Scope: Fokus pada kontrak modul, struktur data (Entity), dan tanggung jawab utama. Asumsi implementasi menggunakan arsitektur Controller → Service → Repository.

2. Tiering Overview

Tier 1: Must Have

Bagian inti dari platform. Aplikasi tidak dianggap "siap" tanpa ini.

  • Auth, User & Role, Generic CRUD, File Infra, Notification, Audit Log.
Tier 2: Recommended

Sering dibutuhkan di aplikasi modern, idealnya tersedia tapi bisa dinonaktifkan.

  • Settings, Organization/Team, Activity Feed.
Tier 3: Optional

Pattern siap pakai untuk kebutuhan spesifik/kompleks.

  • Workflow Status, Comment System, Tagging.

3. Tier 1 – Wajib Core

3.1. Auth Module

Mekanisme standar autentikasi dan manajemen credential.

Komponen Inti

  • Entity: UserCredential (passwordHash, salt, lastLoginAt).
  • Security: Access Token (JWT Short-lived) & Refresh Token (Long-lived).
  • Flows: Register, Login, Logout, Forgot Password, Reset Password.

Kualitas Minimal

  • Hashing menggunakan Argon2.
  • Rate limiting pada endpoint Login.

3.2. User & Role Module

Representasi identitas dan kontrol akses.

Komponen Inti

  • Entity: User (id, email, name, status).
  • Role: RBAC Sederhana (e.g., admin, user).
  • Mapping: Relasi Many-to-Many UserRole.

3.3. Generic CRUD Pattern

Standarisasi operasi data, pagination, dan format response.

Konvensi Endpoint

  • GET /resource (List + Pagination)
  • GET /resource/:id (Detail)
  • POST /resource (Create)
  • PUT /resource/:id (Update)
  • DELETE /resource/:id (Delete)

Response Format Standard

// Success Response
{
  "success": true,
  "data": [ ... ],
  "meta": { "page": 1, "total": 100 }
}

// Error Response
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Email is required"
  }
}

3.4. File / Attachment Infrastructure

Standar upload dan penyimpanan metadata file.

Komponen Inti

  • Entity: File (originalName, mimeType, size, url).
  • Storage Abstraction: Interface yang mendukung Local Driver (Dev) dan S3/Object Storage (Prod).

3.5. Notification Infrastructure

Kerangka dasar pengiriman notifikasi (Email first).

Komponen Inti

  • Service: sendEmail(to, templateId, variables).
  • Templates: Reset Password, Email Verification, Welcome User.

3.6. Audit Log Infrastructure

Mencatat jejak keamanan dan governance sistem.

Komponen Inti

  • Entity: AuditLog (actorId, action, targetType, targetId, metadata, ip).
  • Trigger: Otomatis mencatat saat Login (sukses/gagal) dan mutasi data sensitif.

4. Tier 2 – Sangat Disarankan Rec

4.1. Settings & Configuration Module

Memisahkan konfigurasi dinamis dari hardcoded code.

  • SystemSetting: Key-value store global (misal: Maintenance Mode toggle).
  • UserPreference: Setting per user (misal: Theme, Notif Preference).

4.2. Organization / Team Module

Mendukung model multi-tenant (SaaS/B2B).

  • Organization: Entity grup perusahaan.
  • Membership: Relasi User ke Organization dengan role spesifik (Owner, Member).

4.3. Activity / Timeline Module

Feed aktivitas yang ramah manusia (beda dengan audit log teknis).

  • Feed: "User A mengomentari Order #123", "Status berubah jadi Done".

5. Tier 3 – Opsional / Advanced Opt

5.1. Generic Status / Workflow

Pola untuk entity dengan state machine yang kompleks (Draft → Review → Approved → Published).

5.2. Comment / Discussion

Sistem komentar yang bisa ditempel ke entity apa saja (Polymorphic association).

5.3. Tag / Label System

Sistem grouping fleksibel menggunakan tagging.

6. Konvensi Umum

Aturan main saat membuat feature module baru:

6.1. Penamaan

Module name lowercase (auth, users). Endpoint URL selalu plural (/api/v1/users).

6.2. Pola Layering

Alur data wajib searah: Routes → Controller → Service → Repository.

6.3. Kontrak API

Gunakan Error Code terstandar (misal: AUTH_INVALID_CREDENTIALS, PERMISSION_DENIED) agar frontend mudah memproses error.

7. Checklist Implementasi

Gunakan daftar ini untuk memvalidasi kelengkapan fitur starterkit sebelum release.

Tier 1 – Wajib

  • Auth: Login, Logout, Reset Password, Secure Token (JWT).
  • User: Entity User & Role, CRUD Admin, Middleware Auth.
  • CRUD: Base Service Pattern, Pagination, Standard JSON Response.
  • File: Upload endpoint, Validation (size/mime), Storage logic.
  • Notif: Email service wrapper, Template dasar.
  • Audit: Entity Log, Hooks pada critical event.

Tier 2 – Recommended

  • Settings: System config CRUD.
  • Org/Team: Team creation & membership logic.
  • Activity: Public activity feed implementation.

Tier 3 – Optional

  • Workflow: State transition logic.
  • Comments: Polymorphic comment system.
  • Tags: Tagging logic.
